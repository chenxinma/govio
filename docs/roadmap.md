# Roadmap

## Phase 1: 结构定义层 ✅ 已完成

### 本地图数据库迁移
**目标：** 构建结构定义层的图模型，支持元数据、数据标准、表关系的存储与查询。

**已完成：**
- [x] 实现 `NetworkXGraph`（GML 文件读写、schema 内省）
- [x] 实现 `FalkorDBGraph`（Cypher 查询、schema 内省）
- [x] 实现 CSV → GML 生成流水线（`gen_networkx.py`）
- [x] 实现 `RelationshipLoader` 支持表间关系（RELATES_TO）
- [x] 实现 `Recommender` 数据标准推荐（k-NN 协同过滤）
- [x] CLI 工具 `metadata` / `gml_generate` 稳定可用

**双后端策略：**
- **NetworkX**：轻量嵌入式，无需外部服务，适合本地开发和离线场景。当前 skill 查询脚本（`scripts/query.py`）基于此实现。
- **FalkorDB**：Cypher 声明式查询，LLM 对 Cypher 的问答生成质量优于 NetworkX 的 Python 代码生成。保留用于对查询表达力有要求的场景。

**遗留：**
- [ ] GML 文件体积随数据增长后的加载性能优化

## Phase 2: 数据内容探查层（进行中）

### MCP 数据集服务
**目标：** 构建 MCP Server，让 agent 在受控安全边界内对数据内容进行多轮探查、比对和检核。

**背景：** 结构定义层的图模型（元数据、数据标准、表关系）解决了「数据是什么」的问题。但实际数据治理中，对数据内容本身的探查同样必要——多数据集比对、数据质量检核、异常发现。这类探查是多轮的，轮次间的中间数据集需要被管理和传递。

**设计决策：** 采用 MCP 而非文件系统存储（详见 [design-thoughts.md](design-thoughts.md)）。核心原因：MCP 可精确控制 agent 的数据可见性（统计摘要 vs 原始数据），文件存储方案的安全边界构建成本过高。

#### 2.1 MCP Server 基础设施
- [ ] 搭建 MCP Server 项目骨架（Python, FastMCP）
- [ ] 定义核心 Tool：数据集注册、列表、移除（生命周期管理）
- [ ] 定义核心 Tool：数据集统计摘要（行数、列类型分布、空值率、唯一值数等）
- [ ] 定义核心 Tool：数据质量检核（规则定义 → 执行 → 返回结果摘要）
- [ ] 定义核心 Tool：数据集比对（schema diff + 统计 diff）
- [ ] 中间数据集管理：探查产生的子集/聚合结果可注册为新数据集，供下一轮使用
- [ ] 数据访问策略：默认仅暴露统计摘要，原始数据访问需显式授权或限行/限量

#### 2.2 Skill 层
- [ ] 设计数据内容探查的 skill 集合（注册数据集、执行检核、比对分析、异常定位等）
- [ ] 与现有 govio skill 整合：结构探查 + 内容探查的统一入口

#### 2.3 与图模型联动
- [ ] MCP 中的数据集与图模型中的 PhysicalTable 节点建立关联
- [ ] 利用图关系（如 RELATES_TO）自动推荐可比对的数据集组合
