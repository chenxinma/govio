# Govio 设计思考

## 起源

Govio 最初从一个独立的数据治理问答应用中剥离出来。核心想法是：在 Claude Code 或 OpenCode 这样的 AI 编码环境中，以 skill 的形式，基于元数据、数据标准等通过图构建的关系进行数据治理领域的问答。

第一阶段的探索集中在**数据模型和结构定义层面**——比对数据在定义上的差异和发现隐含关联。这构成了 Govio 图模型的基础（PhysicalTable、Col、Standard 等节点及其关系）。

## 从结构探查到内容探查

在实际的数据治理过程中，仅停留在模型和结构层面是不够的，对**数据内容本身**的探查同样必要。典型场景是围绕多个数据集进行比对和检核。

这种探查行为的特点是**多轮执行**——每一轮分析会产生中间结果数据集，这些数据集需要在轮次之间传递下去，作为下一轮的输入。因此需要一个机制来管理这些中间态的数据集。

## 两种方案的比较

### 方案一：MCP 服务

构建一个 MCP 服务程序，统一管理数据集，提供：
- 数据集注册与生命周期管理
- 访问接口（查询、统计摘要）
- 统计检核能力（数据质量规则校验）

**优势**：MCP 可以精确控制 agent 能够访问的内容边界。例如仅提供统计结果（行数、空值率、分布等），而非直接暴露原始数据内容。这对数据安全是一个重要的保障——agent 在不需要看到原始数据的情况下就能完成分析任务。

### 方案二：Iceberg 文件存储

将中间数据集以 Apache Iceberg 表格式作为文件存放在本地文件系统中，让 agent 直接通过文件系统访问。

**优势**：Iceberg 提供了版本化的表格式，天然支持时间旅行和 schema 演进。

**问题**：文件存储的方式对数据安全的控制需要更加复杂的处理。一旦 agent 拥有文件系统访问权限，就难以限制它能看到什么——原始数据、敏感字段都可能被直接读取。需要额外构建一整套访问控制层。

## 最终选择

选择了 **MCP 服务方案**，通过一系列 skill 来支撑操作。具体来说：

- MCP server 负责数据集的托管和提供受控的访问接口
- Skill 层定义 agent 的操作语义（注册数据集、执行检核、比对分析等）
- Agent 通过 skill 调用 MCP 工具，始终在安全边界内完成数据探查

这种分层设计将「能做什么」和「能看什么」分离开来：skill 决定操作意图，MCP 决定数据可见性。
