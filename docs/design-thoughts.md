# Govio 设计思考

## 起源

Govio 从一个独立的数据治理问答应用里剥离出来。想法很简单：让 AI 编码环境（Claude Code、OpenCode）通过 skill，基于元数据构建的图关系来回答数据治理问题。

第一阶段关注的是**模型和结构**——比对定义差异、发现隐含关联。PhysicalTable、Col、Standard 这些节点和关系，就是那时候定义的。

## 从结构到内容

光看模型和结构不够。实际的数据治理需要对**内容本身**做探查，比如比对多个数据集、做检核。

这类工作往往是多轮的：每轮分析产生中间结果，传给下一轮继续。所以需要管理这些中间数据集。

## 两个方案

### 方案一：MCP 服务

建一个 MCP 服务统一管理数据集：
- 数据集注册和生命周期管理
- 访问接口（查询、统计摘要）
- 统计检核能力

MCP 能精确控制访问边界。比如只给统计结果（行数、空值率、分布），不暴露原始数据。这对数据安全很关键——agent 不需要看到原始数据也能完成分析。

### 方案二：Iceberg 文件存储

把中间数据集以 Iceberg 表格式存在本地文件系统。

Iceberg 有版本化，支持时间旅行和 schema 演进。这是优势。

但文件存储对数据安全的控制很麻烦。agent 有文件系统权限后，能读原始数据、敏感字段。得额外建一套访问控制层。

## 最终选择

选了 **MCP 服务**，用 skill 支撑操作：

- MCP server 托管数据集，提供受控访问接口
- Skill 定义操作语义（注册数据集、执行检核、比对分析）
- Agent 通过 skill 调用 MCP 工具，在安全边界内完成探查

「能做什么」和「能看什么」分开：skill 决定操作意图，MCP 决定数据可见性。
