## 指标设计

核心思路是**把指标拆解为“业务概念”和“计算逻辑”两部分，用图的形式表达其依赖关系**。这样能清晰展现指标之间的血缘和计算路径。

下面是一个通用的本体模型设计方案：

**1. 核心概念映射**

-   **指标**：作为核心节点，包含名称、编码、业务定义、单位、数据来源等基础属性。
-   **原子/派生指标**：可通过标签区分。原子指标直接来自事实，派生指标则通过**计算节点**定义。
-   **计算节点**：一个独立的节点，专门描述计算公式，例如 `(聚合函数)`、`(算术运算)`、`(比率)`。这样能解耦指标与具体计算。
-   **修饰词/维度**：如“最近7天”、“美国”等，用**边**连接到指标，表示该指标是在特定限定条件下的计算结果。

**2. 关键设计原则**

-   **指标与计算解耦**：`(指标A)` 通过 `(聚合)` 指向 `(原子指标)` 和 `(维度)`。这样，修改计算逻辑只需更新计算节点，而不影响指标的定义。
-   **依赖关系显式化**：利用图数据库优势，用 `依赖` 边连接派生指标及其依赖的原子指标或其他派生指标，形成清晰的血缘关系图。
-   **维度的复用**：将维度（如时间、地域、产品）建模为独立节点，多个指标可以共享，避免数据冗余。

**3. 数据模型示例**

```cypher
// 原子指标
(:Metric {name: “订单总额”, type: “原子”, data_type: “Decimal”, unit: “元”})

// 维度
(:Dimension {name: “订单状态”, values: [“已完成”, “已取消”]})
(:Dimension {name: “时间”, granularity: “日”})

// 计算节点
(:Calculation {type: “SUM”, expression: “sum(order.amount)”})

// 派生指标
(:Metric {name: “已完成订单总额”, type: “派生”, business_definition: “... ...”})

// 构建关系
(原子指标)-[:USES_TABLE]->(:Table {name: “fact_orders”})   // 关联源表
(原子指标)-[:REFERS_COLUMN]->(:Column {name: “amount”})   // 关联字段
(派生指标)-[:CALCULATED_BY]->(计算节点)
(计算节点)-[:BASED_ON]->(原子指标)
(派生指标)-[:FILTERED_BY]->(:Filter {condition: “status = ‘已完成’”})
(派生指标)-[:GROUPED_BY]->(维度)   // 按某维度聚合
```

**4. 存储的具体实现建议**

-   **节点标签**：用 `Metric`、`Dimension`、`Calculation`、`Table`、`Column` 等区分不同实体。
-   **属性存储**：将指标定义的静态信息（名称、负责人、版本、业务口径）存在 `Metric` 节点的属性中。
-   **血缘关系**：用 `DERIVED_FROM`、`BASED_ON`、`FILTERED_BY` 等边，表达指标间或指标与基础数据间的依赖。
-   **版本管理**：可以为每个 `Metric` 节点增加 `version` 属性和 `SUPERSEDES` 边，来管理指标定义的变更历史。

**5. 查询与维护优势**

-   **全链路血缘分析**：`MATCH p=(:Metric)-[:BASED_ON|DERIVED_FROM*1..5]->(:Metric) RETURN p` 能快速找出指标上下游依赖。
-   **影响分析**：修改原子指标时，能快速查询出所有受影响的上层派生指标。
-   **自动推导可计算性**：如果指标依赖的所有上游指标和物理表都存在，则可视为“可计算”状态。

**6. 与其他方案对比**

-   **关系型数据库**：通常用两张表（指标基础信息表 + 指标依赖关系表），查询依赖路径需要递归CTE，复杂且低效。
-   **图数据库**：天然适合表达复杂的依赖网，查询多层血缘关系直观且高效。

总的来说，把指标定义放入图数据库，最大的价值在于用图结构天然地**展现指标间的血缘和计算依赖**。这对数据治理、指标口径的统一管理和根因分析很有帮助。你可以从核心的“指标”、“计算”、“维度”三类节点开始，逐步细化。
